# Phase 2 で学んだこと

## 1. 判別可能ユニオンで成功と失敗を表現できる

`Result` 型で、

* 成功
* 失敗

を同じ戻り値の中で表現できるようになった。

大事なのは、
**例外に頼らず、関数の戻り値として失敗を扱える**
ようになったことです。

今回の学びは「判別可能ユニオンの作り方」だけでなく、
**ドメイン操作の成功/失敗を型で表現するやり方** です。

---

## 2. 独自エラー型は `Error` 継承 class で作れる

独自エラーは `Error` を継承した class として作れる。

そこでやったことは主にこれです。

* `super(...)` でメッセージを渡す
* `name` を設定する
* 必要に応じて `Object.setPrototypeOf(this, new.target.prototype)` を入れる

ここでのポイントは、
**単なる文字列ではなく、「意味のある失敗型」を作れたこと** です。

---

## 3. 独自エラー型を class にする利点は `instanceof` で絞り込めること

これはその理解でいいです。

たとえば失敗時に、

* これはタイトル不正なのか
* これは状態遷移不正なのか

を `instanceof` で判定できるようになります。

つまり、class にしておくと
**エラーの種類を実行時に区別しやすい**
という利点があります。

ただし補足すると、
「独自エラー型は必ず class でなければならない」というより、
**今回の `Result<T, E extends Error>` 設計と相性がいいから class が自然**
という理解がより正確です。

---

## 4. prototype チェーンを少し意識する必要がある

ここは「なんとなく」で十分です。
今の段階では深追いしなくていいです。

理解としては、

* `Error` 継承の独自 class を作るとき
* 実行時にそのインスタンスを正しくその class として扱いたい
* そのために prototype 周りを整えることがある

くらいで大丈夫です。

つまり今必要なのは、
**なぜ `Object.setPrototypeOf(...)` を入れることがあるのかをざっくり知ること**
であって、prototype 理論を全部理解することではないです。

---

## 5. オブジェクトを破壊せずに新しい値を返す更新の書き方

ここも大事です。

今回 `completeTask` / `reopenTask` でやったのは、
元の `task` を直接変更するのではなく、

* 元の値を展開し
* 変更したい部分だけ差し替えた
* 新しい Task を返した

という形です。

学んだのは単なる記法というより、
**破壊的変更を避けて、新しい値として返すやり方** です。

これは TypeScript というより、
後のテストや状態管理でも効く考え方です。

---

## 追加で入れてよい学び

## 6. ドメインルールを関数に閉じ込める考え方

Phase 2 では、

* 完了できる条件
* 再開できる条件
* タイトルが有効な条件

を、呼び出し側に散らさずに
**ドメイン関数の中に集約する**
ようになりました。

これはかなり重要です。

つまり今回学んだのは型の文法だけではなく、
**ルールを直接代入ではなく関数に閉じ込める設計** です。

---

## 7. `createTask` は「生成」だけでなく「失敗しうる生成」になった

Phase 1 では `createTask` は単純生成でしたが、
Phase 2 では **バリデーションに失敗する可能性がある関数** になりました。

ここで学んだのは、
**ドメインの生成処理は常に成功するとは限らない**
ということです。

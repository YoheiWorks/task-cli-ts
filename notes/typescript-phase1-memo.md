# TypeScriptメモ: Phase 1 で学んだこと

## 目的
タスク管理CLIアプリの最初のドメイン実装を通して、
TypeScriptを「文法」ではなく「データ構造と責務を安全に表す道具」として使い始めた。

今回の実装対象は、次のような最小の Task ドメインだった。

- TaskId
- TaskTitle
- TaskStatus
- Task
- createTask

---

## 今回学んだこと

## 1. type でドメインの名前を付けられる
JavaScript では文字列やオブジェクトをそのまま使うことが多いが、
TypeScript では `type` を使って「これは何の値なのか」に名前を付けられる。

例:
- TaskId
- TaskTitle
- TaskStatus
- Task

ここで大事なのは、
最初から高度な型を作ることではなく、
**業務上の概念に名前を与えること**。

### 気づき
- `string` をそのまま使うより、`TaskTitle` のように名前を付けたほうが意図が読める
- ただし `type TaskTitle = string` は、今の段階では「意味づけ」であって、強い区別ではない
- それでも設計上の読みやすさには価値がある

---

## 2. オブジェクト型で Task の構造を表せる
Task は単なる文字列ではなく、複数のデータを持つまとまり。
TypeScript ではオブジェクト型でその形を表せる。

Task が持つ最小データは次の3つでよかった。

- id
- title
- status

### 気づき
- 今必要なデータだけを持たせるのが大事
- `createdAt` や `priority` などは、必要になるまでは入れない
- 型は「何を持つか」をはっきりさせるために使う

---

## 3. Literal 型で取りうる値を固定できる
`TaskStatus` はただの `string` ではなく、
決まった文字列だけを許したい。

そのときに使うのが Literal 型。

考え方:
- `"open"` は単なる文字列ではなく、特定の1値を表す型にもなる
- `"completed"` も同じ

### 気づき
- 「どんな文字列でもよい」ではなく、「この値だけ許す」を型で表せる
- 状態のように候補が決まっているものと相性がいい

---

## 4. Union 型で複数の候補を表せる
`TaskStatus` は `"open"` または `"completed"` のどちらか。

こういう「候補の集合」は Union 型で表せる。

### 気づき
- 状態や種別のような値に向いている
- `string` よりずっと安全
- 予定外の値を入りにくくできる

---

## 5. 関数にも型を付ける
`createTask` では、引数と戻り値に型を持たせる考え方を学んだ。

見るべき点は2つ。

- 何を受け取るか
- 何を返すか

### 気づき
- 関数の型を書くと、役割がかなり明確になる
- `createTask` は「タイトルを受け取って Task を返す」関数として読める
- TypeScript では、関数が設計の入口になることが多い

---

## 6. createTask で生成責務をまとめる
Task を作るたびに呼び出し側で

- id を生成する
- status を `"open"` にする
- Task オブジェクトを組み立てる

とやると、責務が散らばる。

そこで `createTask` にまとめる設計にした。

### 気づき
- 生成処理を1箇所に寄せると、初期状態がぶれにくい
- 呼び出し側が「Task をどう作るか」を知らなくてよくなる
- TypeScript の型は、この責務分離を読みやすくするためにも使える

---

## 7. 型注釈と型推論の使い分け
TypeScript は全部を明示しなくても、文脈から型を推論してくれる。

ただし、学習初期では次の場所は明示する価値が高い。

- ドメイン型の定義
- 関数の引数
- 関数の戻り値

### 気づき
- 何でも明示すればよいわけではない
- 逆に、重要な境界は明示したほうが設計が読みやすい
- 型推論に任せる場所と、明示する場所を分ける意識が必要

---

## 8. 型名は短すぎると弱い
最初は `Id`, `Title`, `Status` のような名前でも書ける。
ただし、実務では曖昧になりやすい。

そこで `TaskId`, `TaskTitle`, `TaskStatus` のように、
対象を名前に含めるほうがよいと判断した。

### 気づき
- 型名は短いことより、文脈が分かることが大事
- ファイルや概念が増えたときに差が出る
- TypeScript では「型の強さ」だけでなく「名前の明確さ」も設計の一部

---

## 9. export は公開したい概念を外に出すために使う
ドメインの型や関数を他のファイルから使うには export が必要。

今回の段階では、次のような概念は外で使う可能性が高い。

- Task
- TaskId
- TaskTitle
- TaskStatus
- createTask

### 気づき
- export は単なる文法ではなく、公開APIの境界
- 今は分かりやすさ優先で全部 export してよい
- 将来は「公開するもの」と「内部だけで使うもの」を分ける意識が必要

---

## 10. 型と実装責務は別で考える
今回気になった論点として、
「ID生成は TaskId 側に持たせるべきでは？」というものがあった。

ここで分かったのは、
**型の意味** と **生成の責務** は別に考える必要があるということ。

### 今の理解
- `TaskId` はまず「Task の識別子という意味」を表す型
- IDをどう生成するかは、今は `createTask` に置いてよい
- 生成戦略の差し替えが問題になるのは、もう少し後のフェーズ

### 気づき
- 型に何でも載せればよいわけではない
- 今困っていない問題まで先回りしすぎない
- 責務分離は、今のフェーズに合った粒度で行う

---

## 今の時点でまだ学んでいないもの
まだ入っていない、または本格的には扱っていないもの。

- バリデーション
- Result 型
- 判別可能 Union
- Narrowing
- class ベースの設計
- interface
- ジェネリクスの活用
- Zod
- Vitest
- Commander.js
- npm workspaces

これは未習得というより、
**まだその段階ではないから使っていない** と理解する。

---

## 今の設計方針
現時点では次の方針で十分。

- Task は最小データだけ持つ
- 状態は Union 型で表す
- 生成は `createTask` にまとめる
- 型名は文脈付きにする
- 難しい型テクニックはまだ使わない

---

## ここまでの理解を一言でまとめる
TypeScript は、
「厳密な型パズルをするためのもの」ではなく、
**ドメインの概念、データ構造、責務の境界を明確にするための道具** として使い始めた。

---

## 次に学ぶこと
Phase 2 では、次の論点に進む。

- 完了 / 再オープンのルール
- タイトルの妥当性
- 失敗をどう表現するか
- Result 型
- ドメインエラー

ここからは「値を表す型」だけでなく、
**成功と失敗を型で扱う** 段階に入る。

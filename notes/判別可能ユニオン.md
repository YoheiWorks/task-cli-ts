判別可能ユニオンは、**複数の型のどれかを表す union 型に対して、共通の「見分け用プロパティ」を持たせる設計**です。TypeScript はそのプロパティの値を見て、分岐の中で型を絞り込めます。公式ドキュメントでは、`kind: "circle" | "square"` のような**共通プロパティ + リテラル型**の形が典型例として紹介されています。 ([TypeScript][1])

たとえば、次のような型です。

```ts
type Shape =
  | { kind: "circle"; radius: number }
  | { kind: "square"; sideLength: number };
```

この `kind` が判別子です。`kind === "circle"` と判定すると、その分岐では TypeScript が「これは circle 側だ」と理解し、`radius` を安全に使えるようになります。これは TypeScript の **narrowing（型の絞り込み）** と **control flow analysis（制御フロー解析）** によるものです。 ([TypeScript][2])

## どういう仕組みか

TypeScript の union 型は、そのままだと「どの型か未確定」なので、共通でないプロパティにはそのままアクセスできません。ですが、`if` や `switch` で判別子をチェックすると、TypeScript がその条件に応じて型を狭めてくれます。 ([TypeScript][2])

```ts
function area(shape: Shape): number {
  if (shape.kind === "circle") {
    return Math.PI * shape.radius ** 2;
  }

  return shape.sideLength ** 2;
}
```

このように、**実行時の条件分岐を使って静的に型安全を得る**のが判別可能ユニオンの強みです。 ([TypeScript][2])

## なぜ便利なのか

便利な理由は、**「状態ごとに持つデータが違う」ことを自然に表現できる**からです。
たとえば、通信状態なら `"loading" | "failed" | "success"`、処理結果なら `"ok" | "err"` のように表せます。TypeScript公式でも、文字列リテラル型は「特定の決まった値だけを許す」ために使えると説明されています。 ([TypeScript][3])

```ts
type NetworkState =
  | { state: "loading" }
  | { state: "failed"; code: number }
  | { state: "success"; response: string };
```

この設計にすると、`state` を見れば「どんなデータが入っているはずか」が分かります。
つまり、**if文の条件が、そのまま型安全な設計になる**わけです。 ([TypeScript][1])

## どんなときに使うべきか

使うべきなのは、**1つの値が複数の状態のどれかになり、その状態ごとに持つプロパティが違うとき**です。
代表例は次のようなものです。

* APIの成功 / 失敗
* ローディング / 成功 / 失敗
* 図形の種類ごとのデータ
* フォーム入力の状態ごとの値

たとえば `Result` ならこうです。

```ts
type Result<T, E> =
  | { kind: "ok"; value: T }
  | { kind: "err"; error: E };
```

```ts
function unwrap<T, E>(result: Result<T, E>): T {
  switch (result.kind) {
    case "ok":
      return result.value;
    case "err":
      throw result.error;
  }
}
```

このような「成功なら `value`、失敗なら `error`」という関係は、判別可能ユニオンと非常に相性が良いです。 ([TypeScript][1])

## 文字列がよく使われる理由

判別子には文字列リテラルがよく使われます。
TypeScript のリテラル型には文字列・数値・boolean がありますが、判別可能ユニオンでは特に**文字列**が一般的です。理由は、状態が増えても読みやすく、`switch` と相性が良いからです。 `"ok" | "err"` や `"loading" | "success" | "failed"` のように増やしやすいのも利点です。 ([TypeScript][3])

```ts
type Result<T, E> =
  | { success: true; value: T }
  | { success: false; error: E };
```

このように boolean でも作れます。
ただ、boolean は 2 状態までしか表しにくいので、将来状態が増える可能性があるなら文字列のほうが柔軟です。これは仕様上の必須条件ではなく、設計上のベストプラクティスです。 ([TypeScript][3])

## 使うべきではないケース

判別可能ユニオンは便利ですが、**単に「同じ形のデータ」を表したいだけのとき**には向きません。
たとえば、どのケースでも同じプロパティしか持たないなら、普通の interface や type alias のほうがシンプルです。TypeScript の object type は、オブジェクトの形を表す基本手段です。 ([TypeScript][4])

また、状態の違いを表したいのに、判別子を持たせずにただ union にするだけだと、絞り込みがしづらくなります。

```ts
type Bad =
  | { value: number }
  | { error: Error };
```

これでも型としては作れますが、`value` 側か `error` 側かを見分ける共通の判別子がないので、分岐が分かりにくくなりやすいです。判別可能ユニオンの良さは、**明示的な判別子で絞り込めること**にあります。 ([TypeScript][1])

## `switch` と網羅性チェック

判別可能ユニオンでは `switch` がとても相性が良いです。
さらに、すべてのケースを処理し忘れていないかを `never` で確認する方法があります。TypeScript公式でも、union の可能性をすべて取り除いたあとに残る型として `never` が説明され、`assertNever` によるチェック例が紹介されています。 ([TypeScript][1])

```ts
function assertNever(x: never): never {
  throw new Error("Unexpected object: " + x);
}

function area(shape: Shape): number {
  switch (shape.kind) {
    case "circle":
      return Math.PI * shape.radius ** 2;
    case "square":
      return shape.sideLength ** 2;
    default:
      return assertNever(shape);
  }
}
```

こうしておくと、あとで `triangle` を追加したのに `switch` を直し忘れた、というミスを検出しやすくなります。 ([TypeScript][1])

## 判別可能ユニオンの基本ルール

初心者向けにまとめると、押さえるポイントはこの4つです。

* union 型にする
* 各要素に共通の判別子を持たせる
* 判別子にはリテラル型を使う
* `if` や `switch` でその判別子を見て絞り込む

たとえば `Result` なら、次の形はかなり素直です。

```ts
type Result<T, E extends Error = Error> =
  | { readonly kind: "ok"; readonly value: T }
  | { readonly kind: "err"; readonly error: E };
```

これは、判別可能ユニオン、ジェネリクス、型制約、`readonly` を組み合わせた形です。どれも TypeScript の基本機能の上に成り立っています。 ([TypeScript][1])

判別可能ユニオンは、**「状態ごとに違うデータを、安全に表現するための型の作り方」**です。
TypeScript では、共通の判別子を持たせることで、分岐の中で自然に型が絞り込まれます。`Result`、通信状態、画面状態の表現で特に強力です。 ([TypeScript][2])
